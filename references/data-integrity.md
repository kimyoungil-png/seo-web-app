# データ整合性ルール — 数値捏造禁止の運用

> このファイルは [SKILL.md](../SKILL.md) の「最上位ルール 1」を**実運用レベル**まで具体化したものです。

---

## 大原則

> **MCPツールが返した値だけを書く。返らない値は書かない。**

「だいたい」「一般的には」「業界平均では」を根拠にした数値を本文・候補一覧・構成案・記事ドラフトに**書かない**。

---

## 数値を扱うときの判定フロー

```
そのページに数値を書きたい
        ↓
その数値は当該 run のいずれかの MCP 呼び出し結果に存在するか?
        ├─ Yes → 書く (期間・単位・出典ツール名を併記)
        └─ No  → 「(データなし)」と明示する。書かない。
```

---

## 出典の表記ルール

数値の脇に**出典**を明示する。ユーザーが後から再現できる粒度で書く。

| 書き方 | 例 |
| --- | --- |
| 単位を必ず付ける | `impressions: 1,234` / `ctr: 1.2%` / `position: 7.4` |
| 期間を明記する | `2026-04-22〜2026-05-19` |
| MCPツール名を併記 (詳細レポートで) | `(出典: gsc_position_window)` |
| 集計が組み合わせなら "クロス" と明示 | `(GSC × GA4 名寄せ後)` |

---

## やってはいけない具体例

| NG | 理由 | OK代替 |
| --- | --- | --- |
| 「平均CTRはおよそ5%です」 | 推測 | 「対象期間の実CTRは1.2%でした (gsc_low_ctr_pages)」 |
| 「業界平均と比べて低い」 | 業界平均値の出典がない | 「同KWでの自社過去30日と比べて0.8pt低い」(両方MCP実数) |
| 「impressions: 〜1000」(範囲ぼかし) | 推測の温床 | 「impressions: 837」 |
| ランキング表で空セルを「-」で埋めずに見栄えで埋める | データ捏造 | 「(GA4側データなし)」と明示 |
| 競合記事に「PVは月50万」と書く | 観測不能なデータ | 書かない |

---

## 「(データなし)」の表記

- セル/箇条書き: `(データなし)` または `—`(本文中なら必ず明示文を添える)
- 比較表で片方欠損: `GA4: 38% / GSC: (データなし)`
- ページ全体で欠損: 「対象期間においてGA4の対応データを取得できませんでした。」

---

## URL 名寄せルール

GSCとGA4でURLが食い違うケースは頻発する。次の正規化規則を適用する:

1. プロトコル統一 (`http` → `https` を許容、`https` 側に寄せる)
2. 末尾スラッシュ統一 (例: 末尾スラッシュあり側に寄せる)
3. クエリパラメータ除去 (`?utm_*` 等)
4. フラグメント除去 (`#section1` を落とす)
5. 大文字小文字: ホスト部は小文字化、パス部はそのまま

正規化しても突き合わせ不可な場合: **片側欠損として扱う** (推測補完しない)。

## 期間既定値

| ツール | 既定期間 |
| --- | --- |
| GSC 系 | 直近28日 |
| GA4 系 | 直近28日 |

ユーザーが期間を指定した場合は、その期間を `run.json` の `period` に保存し、以降のすべての出力に同じ期間を明記する。**run中に期間を勝手に変えない**。

---

## health_check で必ず確認すること

`/seo:find-keywords` 冒頭で `mcp__ictgrowthhacker-analytics__health_check` を呼び、以下を確認:

- `ga4_ok: true`
- `gsc_ok: true`

どちらかが `false` の場合は**処理を停止**し、ユーザーに不足項目を報告する。
